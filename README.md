# Локальный fintech demo prototype

Интерактивное локальное приложение-прототип платежной платформы. Оно показывает роли, модули, статусы, операции, балансы, комиссии, апелляции, уведомления, журнал событий и API-демо на мок-данных.

Приложение не подключает реальные платежные сервисы, не выполняет реальные финансовые операции и не содержит реальных персональных данных.

## Что демонстрирует приложение

- Как мерчант создает платежный ордер.
- Как платформа назначает реквизиты и провайдера.
- Как оператор меняет статусы операции.
- Как успешный ордер меняет баланс.
- Как мерчант создает выплату.
- Как финансовый менеджер подтверждает или отменяет выплату.
- Как спорная операция замораживает часть баланса.
- Как создается и рассматривается апелляция.
- Как уведомления и журнал событий фиксируют изменения.
- Как может выглядеть API-контур без реальных внешних интеграций.

## Быстрый запуск

```bash
npm install
npx prisma migrate dev
npm run seed
npm run dev
```

После запуска откройте браузер:

```text
http://localhost:3000
```

Если порт 3000 занят, можно запустить так:

```bash
npm run dev -- -p 3060
```

## Сброс базы и повторное заполнение

Полный сброс SQLite-базы:

```bash
npx prisma migrate reset
```

Повторное заполнение демо-данными:

```bash
npm run seed
```

Если на Windows Prisma сообщает, что SQLite-файл не существует, обычно достаточно повторить `npx prisma migrate dev`. В текущем проекте миграция уже создана в `prisma/migrations`, поэтому команда должна синхронизировать схему штатно.

## Роли

- Администратор платформы: видит всю систему, мерчантов, провайдеров, ордера, выплаты, балансы, события и интеграции.
- Мерчант: видит свои ордера, выплаты, баланс, реквизиты, уведомления и API-настройки.
- Оператор: видит операции, статусы, реквизиты, спорные ситуации и задачи ручной обработки.
- Финансовый менеджер: видит балансы, комиссии, выплаты, холды и финансовые события.
- Support / апелляции: видит обращения, спорные операции, комментарии, решения и историю апелляций.

Переключатель ролей находится в верхней панели. Авторизация намеренно не реализована, чтобы демо можно было быстро показывать и исследовать.

## Модули

- Главный дашборд: KPI, оборот, доступный и замороженный баланс, активные риски, последние события.
- Кабинет мерчанта: баланс конкретного мерчанта, последние ордера, API key и callback URL.
- Админ-панель: обзор мерчантов, провайдеров, пользователей и рисков.
- Операционный кабинет: очередь ордеров, активные апелляции и реквизиты внимания.
- Заказы / ордера: таблица ордеров, поиск, фильтр, создание ордера, смена статусов, перевод в спор.
- Детали ордера: параметры, timeline жизненного цикла, движения баланса, связанные апелляции.
- Выплаты: создание выплаты, подтверждение и отмена.
- Реквизиты: тип, банк, лимиты, статус, количество связанных операций.
- Балансы: доступные средства, холды, комиссии и история изменений.
- Комиссии: правила комиссий, ставки мерчантов и провайдеров.
- Апелляции: список обращений, комментарии, взятие в работу и решение.
- Уведомления: события для ролей.
- Журнал событий: аудит ключевых действий.
- Интеграции: демо-провайдеры, статусы, комиссии, доступность и тестовый режим.
- API-демо: примеры создания ордера, webhook, ответа API и смены статуса.
- Карта процессов: визуальные схемы ордера, выплаты, апелляции, баланса и участников.
- Сценарии: 10 кликабельных сценариев, которые меняют данные в SQLite.
- Обучающий раздел: простые объяснения логики системы.

## Интерактивные сценарии

- Мерчант создает платежный ордер.
- Платформа назначает реквизиты.
- Оператор проводит ордер по статусам.
- Успешный ордер обновляет баланс.
- Мерчант создает выплату.
- Финансовый менеджер подтверждает выплату.
- Спорная операция замораживает баланс.
- Создается апелляция.
- Support рассматривает апелляцию.
- Решение апелляции меняет статус операции и баланс.

Каждый сценарий записывает событие в журнал. Часть сценариев создает уведомления, меняет статусы, двигает баланс или создает апелляции.

## Бизнес-логика статусов

Ордер проходит основной путь:

```text
Создан -> Ожидает оплаты -> Оплачен -> Подтвержден -> Завершен
```

Спорная ветка:

```text
Активный ордер -> Спор -> Апелляция -> Решение -> Подтвержден или Отменен
```

Выплата:

```text
Создана -> Ждет подтверждения -> В холде -> Выплачена или Отменена
```

Апелляция:

```text
Новая -> В работе -> В пользу мерчанта / В пользу платформы / Отклонена
```

## Как меняются балансы

- Завершенный ордер увеличивает доступный баланс мерчанта на сумму за вычетом комиссии.
- Комиссия по ордеру увеличивает комиссионный баланс.
- Создание выплаты списывает сумму и комиссию из доступного баланса.
- Создание выплаты добавляет сумму и комиссию в замороженный баланс.
- Подтверждение выплаты списывает сумму из замороженного баланса.
- Отмена выплаты возвращает сумму из холда в доступный баланс.
- Спорная операция замораживает часть суммы.
- Решение апелляции возвращает холд мерчанту или списывает его в пользу платформы.

## Апелляции

Апелляция связана с конкретным ордером. В ней есть причина, статус, автор, ответственный, комментарии, замороженная сумма и решение.

Support может взять апелляцию в работу, добавить комментарий и закрыть ее в пользу мерчанта или платформы. Решение апелляции влияет на статус ордера и баланс.

## Уведомления и журнал событий

Уведомления отвечают на вопрос: кому нужно обратить внимание на событие.

Журнал событий отвечает на вопрос: что произошло, кто сделал действие, какую сущность изменили и когда.

Фиксируются создание ордера, смена статуса, создание выплаты, заморозка баланса, создание апелляции, решение апелляции, изменение реквизитов и шаги сценариев.

## Seed-данные

Seed-скрипт создает:

- 3 мерчанта.
- 5 пользователей с разными ролями.
- 10 платежных ордеров.
- 5 выплат.
- 5 реквизитов.
- 3 апелляции.
- 5 провайдеров.
- 20 событий журнала.
- 10 уведомлений.
- Балансы, комиссии и разные статусы операций.

Все данные реалистичные, но вымышленные.

## Допущения

- Прототип сделан как локальный монолит Next.js вместо набора микросервисов.
- Роли имитируются переключателем без настоящей авторизации.
- Все суммы считаются в RUB, хотя в интерфейсе показана валютная структура.
- Провайдеры и webhook-события моковые.
- Комиссии упрощены до фиксированных ставок.
- Апелляционные доказательства представлены комментариями, без загрузки файлов.
- API key и callback URL являются демо-полями.
- SQLite используется как локальное хранилище, не как production-база.

## Ограничения

- Нет реальных платежей.
- Нет реальных внешних API.
- Нет настоящей авторизации и разграничения доступа на backend-уровне.
- Нет загрузки файлов и настоящих чеков.
- Нет бухгалтерского учета, налоговой логики и сверки с банками.
- Нет очередей, Kafka/NATS, Redis и фоновых воркеров.
- Нет production-hardening, мониторинга и полноценного RBAC.

## Как смотреть демо

1. Откройте главный дашборд и посмотрите общие показатели.
2. Переключите роль в верхней панели.
3. Откройте `Заказы / ордера` и создайте новый ордер.
4. Проведите ордер по статусам до завершения.
5. Откройте `Балансы` и проверьте историю изменений.
6. Создайте выплату и подтвердите ее.
7. Переведите ордер в спор и создайте апелляцию.
8. Закройте апелляцию и проверьте журнал событий.
9. Откройте `Сценарии`, чтобы пройти весь процесс пошагово.
10. Используйте обучающие страницы для объяснения продукта партнеру или инвестору.
